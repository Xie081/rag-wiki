# Prompt 报告 — 智能知识库管理系统

> 记录为构建 RAG 智能问答系统所设计的 Prompt 模板及其迭代优化过程。

---

## 1. 概述

本系统采用模板化 Prompt 管理，所有提示词存储在数据库 `prompt_templates` 表中，
通过 `PromptTemplateService` 进行变量注入渲染。系统包含两个核心模板：
- **rag-qa**：RAG 问答模板（检索增强生成）
- **summarize**：文档摘要生成模板

所有模板支持 `{variable}` 占位符，运行时由 `PromptTemplateService.render()` 动态替换为实际值。

---

## 2. 模板一：RAG 问答 (`rag-qa`)

### 2.1 用途

用于基于知识库文档内容的 RAG 问答。用户提问后，系统先通过向量检索找到相关文档块，
再将检索结果作为上下文注入 Prompt，由 LLM 根据上下文生成答案。

### 2.2 设计

#### System Prompt

```
你是一个专业的知识库助手。你的任务是严格根据提供的文档内容回答用户问题。

规则：
1. 只使用下面【参考文档】中的信息回答问题
2. 如果文档中没有相关信息，请明确告知用户"该知识库中未找到相关信息"
3. 回答时引用具体的文档名称和段落
4. 保持回答简洁、准确、结构化
5. 使用 Markdown 格式组织回答
```

#### User Prompt

```
【参考文档】
{context}

【用户问题】
{question}

请根据以上参考文档回答用户的问题。
```

#### 变量说明

| 变量 | 来源 | 说明 |
|------|------|------|
| `{context}` | PGVector 向量检索 Top-5 | 格式化为 `【来源N】文档：《文档名》\n内容...` 的引用块 |
| `{question}` | 用户输入 | 原始问题文本 |

### 2.3 设计思路

1. **角色设定**（"专业的知识库助手"）：明确 AI 的身份定位，引导其以专业态度回答
2. **约束边界**（规则 1-2）：防止大模型"幻觉"（hallucination），严格限定回答范围
3. **来源溯源**（规则 3）：要求引用文档名称和段落，让用户可验证答案
4. **输出格式**（规则 4-5）：要求结构化、Markdown 渲染，提升可读性

### 2.4 迭代历史

| 版本 | 问题 | 改进 |
|------|------|------|
| v1 | AI 有时编造信息，引用不存在的文档 | 新增规则 1："只使用参考文档中的信息" |
| v2 | 知识库无相关内容时，AI 仍强行回答 | 新增规则 2：明确告知"未找到相关信息" |
| v3 | 回答过于冗长，缺少结构 | 新增规则 4-5：简洁准确 + Markdown 格式 |
| v4 | AI 引用不够具体 | 在 User Prompt 中给每条上下文加 `【来源N】` 编号 |

### 2.5 示例

**输入：**
- 用户问题："项目的技术栈是什么？"
- 检索到 2 个相关文档块（README.md 的技术栈表格段落）

**输出：**

> 根据参考文档，该项目使用的技术栈如下：
>
> | 层级 | 技术 | 版本 |
> |------|------|------|
> | 后端框架 | Spring Boot | 3.5.14 |
> | 语言 | Java | 21 |
> | 前端框架 | Vue 3 + TypeScript | — |
> | 数据库 | PostgreSQL + PGVector | 16+ |
>
> *参考来源：《README.md》*

## 3. 模板二：文档摘要 (`summarize`)

### 3.1 用途

在文档上传并完成分块处理后，自动调用 LLM 为文档生成一段简洁的中文摘要。
摘要存储到 `documents.summary` 字段，在文档详情页展示。

### 3.2 设计

#### System Prompt

```
你是一个文档摘要专家。请为以下文档内容生成一个简洁的摘要。

要求：
1. 摘要长度控制在 200 字以内
2. 突出文档的核心主题和关键结论
3. 使用清晰的中文表达
```

#### User Prompt

```
【文档标题】{document_title}
【文档内容】{content}

请生成摘要：
```

#### 变量说明

| 变量 | 来源 | 说明 |
|------|------|------|
| `{document_title}` | 文档上传时的文件名 | 帮助 AI 理解文档主题 |
| `{content}` | 文档解析后的纯文本（截取前 3000 字） | 提供文档内容 |

### 3.3 设计思路

1. **角色设定**（"文档摘要专家"）：明确任务类型，引导 AI 生成摘要而非分析
2. **长度控制**（规则 1："200 字以内"）：适应前端卡片展示，避免摘要过长
3. **内容聚焦**（规则 2）：要求突出核心主题和结论，不是简单截取
4. **截断处理**：`DocumentProcessingService` 将长文档截取前 3000 字符，平衡信息量和 token 消耗

### 3.4 迭代历史

| 版本 | 问题 | 改进 |
|------|------|------|
| v1 | 摘要过长（500+ 字），前端展示拥挤 | 新增 200 字限制 |
| v2 | AI 生成英文摘要（英文 PDF） | System Prompt 明确要求"使用清晰的中文表达" |
| v3 | 超长文档摘要质量差 | `DocumentProcessingService` 将 context 截断为前 3000 字 |

## 4. Prompt 工程原则总结

### 4.1 核心原则

1. **角色分配（Role Assignment）**：每条 System Prompt 以角色定义开头，设定 AI 的"身份"和行为模式
2. **约束边界（Constrained Generation）**：通过明确规则限制 AI 的输出范围，防止幻觉和跑题
3. **结构化输出（Structured Output）**：要求 Markdown 格式、字数限制等，提高输出的可用性和一致性
4. **模板化变量注入（Template Variables）**：将动态内容（上下文、用户问题）通过占位符注入，实现 Prompt 与业务逻辑的解耦
5. **渐进式优化（Iterative Refinement）**：通过不断测试发现问题并针对性调整 Prompt

### 4.2 反幻觉策略

RAG 场景中最关键的问题是 LLM 的"幻觉"（编造不存在的信息）。本系统的多层防护：

| 层级 | 策略 | 位置 |
|------|------|------|
| Prompt 约束 | "只使用参考文档中的信息" | System Prompt 规则 1 |
| 检索质量 | PGVector 余弦相似度检索 Top-5 | `RagService` |
| 来源溯源 | 每条回答附带文档名称和片段引用 | `CitationSource` 机制 |
| 明确拒绝 | "该知识库中未找到相关信息" | System Prompt 规则 2 |

## 6. 经验教训与未来改进

### 6.1 经验

1. **Prompt 模板化管理是必要的**：将 Prompt 存储在数据库而非硬编码，便于热更新、A/B 测试和版本管理
2. **中文 Prompt 的语序差异**：英文模板生成的回答偏"翻译腔"，中文模板更自然
3. **上下文窗口有限**：Top-K 策略需要在"信息充分"和"上下文长度"之间权衡，当前 K=5 是较好的平衡点

### 6.2 改进方向

1. **增加引用页码**：在 PDF 解析时保留页码信息，让 AI 引用更精确
2. **动态 Prompt 选择**：根据问题类型（事实查询 / 总结 / 对比）选择不同的 System Prompt
3. **多轮对话支持**：将历史对话纳入 Prompt 上下文，实现追问和澄清
4. **Token 计数监控**：在 `TextChunker` 中基于 token 数而非字符数分块，提高 embedding 质量
5. **调优分块参数**：根据文档类型（PDF vs Markdown）自适应调整 chunk_size 和 overlap
